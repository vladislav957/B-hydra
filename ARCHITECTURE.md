# Архитектура B-hydra

Одноранговая электронная денежная система (P2P electronic cash) на собственной
криптографии. Этот документ — карта всей системы: слои, модель данных,
консенсус, сеть и модель безопасности.

---

## 1. Слои (пакет `b_hydra/`)

Каждый слой опирается только на нижние — снизу вверх:

```
┌───────────────────────────────────────────────────────────────┐
│  Точки входа:  manig.py · cli.py · api.py · P2P.py · mobile_client.py │
├───────────────────────────────────────────────────────────────┤
│  api.py            REST API + веб-обозреватель (explorer.html)   │
│  p2p.py · tcp.py   P2P-сеть: gossip, синхронизация, транспорт    │
├───────────────────────────────────────────────────────────────┤
│  node.py           Узел: UTXO-набор, мемпул, майнинг, валидация  │
├───────────────────────────────────────────────────────────────┤
│  blockchain.py     Блоки, PoW, динамическая сложность, работа    │
│  economics.py      Награда, халвинг, конец эмиссии (3000)        │
│  transaction.py    UTXO-транзакции (входы/выходы), мемпул        │
│  contract.py       Смарт-контракт и эскроу                       │
├───────────────────────────────────────────────────────────────┤
│  wallet.py         Ключи ECDSA secp256k1, адреса, подпись        │
│  merkle.py         Дерево Меркла (двойной SHA-512)               │
│  hashcash.py       Proof-of-work примитив                        │
├───────────────────────────────────────────────────────────────┤
│  hashing.py        Переключаемый движок SHA (pure ⇄ hashlib)     │
│  sha2.py           SHA-256/512 «с нуля» (FIPS 180-4)             │
└───────────────────────────────────────────────────────────────┘
```

**Поток зависимостей:** `sha2 → hashing → {merkle, wallet, transaction,
blockchain, hashcash} → node → {p2p, api}`.

## 2. Хеширование — переключаемый движок

Весь SHA в проекте идёт через `hashing.py`. Движок по умолчанию — **наша
реализация «с нуля»** (`sha2.py`), но хеши побитово равны `hashlib`, поэтому
переключение не влияет на консенсус:

```
            ┌── BHYDRA_PURE_SHA=1 (по умолчанию) → sha2.py   (свой код)
hashing.sha512(x) ──┤
            └── BHYDRA_PURE_SHA=0                → hashlib    (быстро)
```

## 3. Модель данных

### Кошелёк (wallet.py)
```
приватный ключ ──(× G на secp256k1)──▶ публичный ключ ──(хеш)──▶ адрес
   256-бит секрет        ECDSA            04||X||Y         "BHY"+Base58
```
- Адрес = `BHY` + Base58( `0x1f` || RIPEMD160(SHA-512(pub)) || checksum ).
- Подпись: ECDSA secp256k1 над SHA-512 сообщения, формат `r||s` (low-s).

### Транзакция UTXO (transaction.py)
Как в Bitcoin — входы тратят прошлые выходы, выходы блокируются на адрес:
```
Transaction
 ├─ vin:  [ TxInput(txid, index, public_key, signature), … ]   ← ссылки на UTXO
 ├─ vout: [ TxOutput(amount, address), … ]                     ← новые выходы
 └─ txid = SHA-512(chain_id + vin-outpoints + vout + timestamp)
            (chain_id вшит в подпись → защита от replay между сетями)

coinbase: один вход «ниоткуда» (NULL_TXID), без подписи — награда майнеру.
```

### Блок (blockchain.py)
```
Block
 ├─ index, previous_hash, timestamp, nonce, difficulty
 ├─ merkle_root = корень Меркла транзакций (двойной SHA-512)
 ├─ data        = [coinbase, tx, tx, …]   (только адреса и суммы)
 └─ hash = SHA-512(index || prev || merkle || timestamp || difficulty || nonce)
```
`difficulty` входит в хеш — её нельзя занизить задним числом.

### Цепочка
```
[Генезис]──prev──[#1]──prev──[#2]──prev──[#3]── …
 (общий для всех узлов, фиксированный timestamp)
```

## 4. Консенсус

- **PoW (как Bitcoin):** майнер перебирает `nonce`, пока хеш как число не станет
  `≤ target` (порог). Меньше target → пропорционально труднее.
- **Сложность пропорциональна числу майнеров:** `target = генезис / N`, где N —
  число разных майнеров в цепочке (`expected_target`). Больше майнеров → труднее,
  меньше → проще. Детерминировано из цепочки → проверяет каждый узел.
- **Выбор главной цепочки:** по **суммарной работе** `Σ (2^512 / target)`
  (`total_work`), а не по длине — длинная цепочка дешёвых блоков не победит.

```
Майнинг блока:
  собрать tx из мемпула → coinbase(награда+комиссии)
        → подобрать nonce (PoW) → добавить в цепочку → разослать сети
```

## 6. P2P-сеть (p2p.py + tcp.py)

### Транспорт
JSON-сообщения поверх TCP с 4-байтовым префиксом длины (лимит 32 МБ).

### Протокол сообщений
```
ping        → pong
hello{host,port}  → peers[…]        (знакомство + обмен пирами)
get_peers   → peers[…]
get_height  → height
get_chain   → chain[…], base_difficulty
transaction{tx, from}  → ack        (+ gossip дальше)
block{block, from}     → ack,height (+ gossip дальше / sync если отстал)
```

### Gossip — распространение по всей сети (multi-hop)
Узел пересылает **новый** блок/транзакцию остальным соседям (кроме источника),
с дедупликацией (`seen_blocks`, `seen_tx`) против петель:

```
A майнит блок
A ──block──▶ B ──block──▶ C ──block──▶ D
              (B переслал)   (C переслал)
            даже если A и D НЕ соединены напрямую — блок дойдёт.
```

### Синхронизация
Новый/отставший узел подтягивает цепочку с наибольшей работой:
```
node.sync():  у кого выше? → get_chain → replace_chain (полная проверка)
```

## 7. Модель безопасности

### Что защищено кодом ✅
| Угроза                                   | Защита                          |
|------------------------------------------|---------------------------------|
| Трата чужих средств                      | ECDSA-подпись + адрес из ключа  |
| Двойная трата / перерасход / подмена суммы | проверка UTXO в блоке/мемпуле  |
| Фальшивый coinbase (печать монет)        | `coinbase ≤ награда + комиссии` |
| Подделка транзакций в чужой цепочке      | полная ревалидация `_validate_chain` |
| Длинная цепочка дешёвых блоков           | консенсус по работе `total_work`|
| Подмена данных блока                     | корень Меркла в хеше            |
| Занижение сложности                      | difficulty в заголовке/хеше     |
| Чужая сеть                               | общий генезис                   |
| Манипуляции временем                     | будущее (+2ч) / ход назад       |
| Replay между сетями/форками              | chain_id в подписи транзакции   |
| XSS в обозревателе                       | экранирование + data-атрибуты, валидация адреса |
| Invalid-curve публичный ключ             | проверка точки на кривой в verify() |
| DoS по памяти                            | лимиты сообщения/тела/блока     |

### Что НЕ лечится кодом ⚠️
- **51%-атака** — свойство PoW: нужна честная вычислительная мощность больше,
  чем у атакующего. На чистом Python хешрейт низкий — это учебная сеть.

## 8. Точки входа

```
python manig.py          полный демо-цикл (кошельки → майнинг → перевод)
python cli.py …          CLI: wallet / init / mine / send / balance / chain
python api.py            REST API + обозреватель  http://<IP>:8000/
python P2P.py --port …   узел распределённой сети (--peer host:port)
python mobile_client.py  эталонный мобильный кошелёк (подпись на устройстве)
pytest                   автотесты
```

```
        ┌──────────── узел B-hydra ────────────┐
HTTP ──▶ api.py ──▶ BHydraNode ◀── p2p.py ◀── TCP (другие узлы)
браузер │           (blockchain,                │
телефон │            UTXO, мемпул)              │
        └──────────────────────────────────────┘
```
