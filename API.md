# B-hydra REST API — для мобильных кошельков

Узел B-hydra отдаёт JSON по HTTP. **Приватный ключ никогда не отправляется на
сервер**: телефон подписывает транзакцию локально и шлёт уже подписанную.

Запуск узла:

```bash
python api.py --port 8000        # http://<IP-компьютера>:8000
```

Открой `http://<IP>:8000/api/info` в браузере телефона (по Wi-Fi) — проверка связи.

---

## Эндпоинты

| Метод | Путь | Описание |
|------|------|----------|
| GET  | `/api/info` | параметры сети, высота, награда за следующий блок |
| GET  | `/api/balance/<address>` | баланс адреса (сумма его UTXO) |
| GET  | `/api/utxos/<address>` | непотраченные выходы адреса (для построения входов) |
| GET  | `/api/chain` | вся цепочка блоков |
| GET  | `/api/mempool` | число неподтверждённых транзакций |
| POST | `/api/transaction` | отправить **подписанную** транзакцию |
| POST | `/api/mine` | добыть блок: `{"miner": "<address>"}` |

### Примеры

```bash
curl http://IP:8000/api/info
curl http://IP:8000/api/balance/BHY...
curl http://IP:8000/api/utxos/BHY...
curl -X POST http://IP:8000/api/mine -d '{"miner":"BHY..."}'
curl -X POST http://IP:8000/api/transaction -d '<signed tx json>'
```

Формат подписанной транзакции (тело `POST /api/transaction`):

```json
{
  "vin": [
    {"txid": "<hex>", "index": 0,
     "public_key": "<hex 130 симв.>", "signature": "<hex 128 симв.>"}
  ],
  "vout": [
    {"amount": 10.0, "address": "BHY<получатель>"},
    {"amount": 39.5, "address": "BHY<сдача-себе>"}
  ],
  "timestamp": 1718000000.0
}
```

## Как подписать транзакцию на телефоне

Всё считается **на устройстве**. Шаги (см. эталон в `mobile_client.py` и `wallet.py`):

### 1. Ключи и адрес
- **Приватный ключ** — случайное 256-битное число (кривая **secp256k1**).
- **Публичный ключ** — `pub = priv × G`, формат несжатый: `0x04 || X(32) || Y(32)` → hex.
- **Адрес**:
  ```
  h       = RIPEMD160(SHA-512(pub_bytes))           # 20 байт
  payload = 0x1f || h
  checksum= SHA-512(SHA-512(payload))[0:4]
  address = "BHY" + Base58(payload || checksum)
  ```

### 2. Канонический payload для подписи
Подписывается и хешируется **только** этот объект (без `public_key`/`signature`):

```json
{"timestamp": <float>,
 "vin":  [{"index": <int>, "txid": "<hex>"}, ...],
 "vout": [{"address": "<str>", "amount": <float>}, ...]}
```

Сериализация — JSON с **сортировкой ключей** (`sort_keys=true`), UTF-8.
Это и есть `signing_payload`. `txid` транзакции = `SHA-512(signing_payload)` (hex).

### 3. Подпись (ECDSA secp256k1 + SHA-512)
```
z = int(SHA-512(signing_payload)) >> (512 - N_bits)   # N_bits = 256
(r, s) = ECDSA_sign(priv, z)        # k — случайное; используем low-s: s ≤ N/2
signature = hex(r, 32 байта) + hex(s, 32 байта)        # 128 hex-символов
```

В **каждый** вход кладём `public_key` и одну и ту же `signature` (один подписант).

### 4. Отправка
`POST /api/transaction` с телом из `to_dict()`. Сервер проверит:
вход ссылается на существующий UTXO; `address(public_key) == адрес расходуемого
выхода`; подпись валидна; сумма входов ≥ суммы выходов. Прошло → транзакция в
мемпуле, подтвердится при ближайшем майнинге.

## Важно про точность сумм (interop)

Сейчас `amount` — число с плавающей точкой, а подпись зависит от **точного**
текста JSON. Разные языки могут по-разному печатать float (`10.0` vs `10`),
и тогда подпись не сойдётся. Для кросс-языкового клиента сериализуй суммы
**идентично** Python (`json.dumps`). На будущее надёжнее перейти на целые
единицы (как сатоши в Bitcoin) — тогда неоднозначности нет.

## Безопасность

- Приватный ключ **не покидает телефон** — на сервер уходят только публичный
  ключ и подпись.
- Сервер по умолчанию слушает локальную сеть. Для публичного доступа поставь
  его за HTTPS-прокси (например, nginx/caddy) — у самого узла TLS нет.
